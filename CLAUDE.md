## Build Commands

This project uses **XcodeGen** to generate the Xcode project from `project.yml`.

```bash
# Generate Xcode project (run this after modifying project.yml)
xcodegen generate

# Open in Xcode
open basenana.xcodeproj

# Build from command line
xcodebuild -project basenana.xcodeproj -scheme BasenanaApp -configuration Debug build

# Run tests
xcodebuild test -project basenana.xcodeproj -scheme DomainTests
xcodebuild test -project basenana.xcodeproj -scheme DataTests
xcodebuild test -project basenana.xcodeproj -scheme FeatureTests
xcodebuild test -project basenana.xcodeproj -scheme StyleguideTests
```

## Architecture

This is a macOS desktop application using **Clean Architecture** with XcodeGen for project management.

### Project Structure

```
basenana/
├── project.yml                 # XcodeGen project configuration (source of truth)
├── Package.resolved            # SPM resolved dependencies
├── Sources/                    # Source root
│   ├── Domain/                 # Domain layer (independent)
│   ├── Data/                   # Data layer (depends on Domain)
│   ├── Feature/                # Feature layer (depends on Domain, Data, Styleguide)
│   │   ├── Entry/              # Feed/group management, WebPage fetching
│   │   ├── Document/           # Document reading, search, masonry layout
│   │   └── Workflow/           # Workflow automation UI
│   └── Styleguide/             # Styleguide layer (independent)
├── Tests/                      # Test directory
│   ├── DomainTests/
│   ├── DataTests/
│   ├── FeatureTests/
│   └── StyleguideTests/
├── basenana/                   # App layer (BasenanaApp target)
│   ├── basenanaApp.swift       # App entry point
│   ├── DI/                     # Dependency injection container
│   ├── Environment/           # Environment configuration
│   ├── Share/                  # Share extension
```

### Layer Structure

| Directory | Layer | Purpose |
|-----------|-------|---------|
| `basenana/` | App | App entry point, DI container, environment config, share extension |
| `Sources/Domain/` | Domain | Entities, AppState, Repository protocols, UseCases |
| `Sources/Data/` | Data | Network clients, REST API, Repository implementations |
| `Sources/Feature/` | Feature | UI modules (Entry, Document, Workflow) |
| `Sources/Styleguide/` | Styleguide | Reusable SwiftUI components and styling |

### Dependency Flow

```
basenana (Executable)
    |
    +-- Domain (independent, no dependencies)
    +-- Data (depends on Domain)
    +-- Feature (depends on Domain, Data, Styleguide)
    +-- Styleguide (independent)
    |
    +-- External: Swinject, Factory, SwiftyBeaver, FeedKit, SwiftSoup, Fuzi, SwiftUIMasonry
```

### Key Patterns

- **Dependency Injection**: `Container.swift` using Swinject and Factory property wrappers
- **State Management**: `StateStore` in Domain layer (`Sources/Domain/AppState/Store/`) using Swift Observation (`@Observable`)
- **Repository Pattern**: Protocols in Domain (`Sources/Domain/RepositoryProtocol/`), implementations in Data (`Sources/Data/Repositories/`)
- **Use Case Pattern**: Protocol + Implementation pairs in Domain (`Sources/Domain/UseCases/`)
- **Protocol-Oriented Design**: Heavy use of protocols for entity models (e.g., `EntryInfo`, `EntryDetail`)
- **Layered Error Handling**: Three-tier enums - `BizError`, `RepositoryError`, `UseCaseError`
- **Tree Structure**: `TreeNode` class for hierarchical navigation with URI-based tracking
